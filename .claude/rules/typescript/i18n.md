# Internationalisation (i18n)

## Scope
These rules apply to all React code in Sites/main. Every piece of user-visible text must go through the translation system — no hard-coded English strings in JSX.

---

## How the System Works

All translation keys and their English strings live in a single file:
```
Sites/main/src/Languages/en-US.json
```

The `LanguageKeys` type is **automatically derived** from that file — it is never hand-edited:

```typescript
// Sites/main/src/Languages/LanguagesType.ts
type EnglishType = keyof typeof localizationEnUs;
type TestType = keyof typeof localizationTeSt;
export type LanguageKeys = TestType | EnglishType;
```

Adding a key to `en-US.json` makes it immediately available as a type-safe `LanguageKeys` value. TypeScript catches invalid or misspelled keys at compile time.

---

## Adding a New Translation Key

Add the key-value pair to `en-US.json` only. Do not add it to any other language file:

```json
{
  "myFeature-myComponent-actionLabel": "Save Changes"
}
```

The key is then immediately usable with full type safety. Do not manually update `LanguageKeys` — it updates itself.

### Key Naming Convention
Keys use kebab-case with a hierarchical prefix that reads from broad to specific:

```
{feature}-{component}-{description}
```

Examples: `audio-control-header`, `activities-schedule-addToSchedule`, `account-type-accessGroup-explainer-Api`

Keep keys descriptive enough that the location and purpose are obvious without reading the value.

## Rendering Translated Text

### Use `<FormattedMessage>` in JSX
When rendering translated text directly in the component tree:

```tsx
<Typography variant="h1">
	<FormattedMessage id="audio-control-header" />
</Typography>

<Button>
	<FormattedMessage id="audio-control-changeSource" />
</Button>
```

### Use the `<Translation>` component for Typography
For translated text that is also a Typography element, use the `Translation` convenience wrapper rather than nesting `FormattedMessage` inside `Typography` manually:

```tsx
// Prefer — cleaner for Typography-wrapped translations
<Translation translationId="audio-view-tab-sources-add-form-Preset-header" variant="h2" sx={{mt: 2}} />

// Avoid — verbose when Translation would do the same
<Typography variant="h2" sx={{mt: 2}}>
	<FormattedMessage id="audio-view-tab-sources-add-form-Preset-header" />
</Typography>
```

### Use `formatMessage` for String Values
When you need the translated text as a string — for HTML attributes, object values, function arguments, or string composition — use `const {formatMessage} = useIntl()`:

```tsx
const {formatMessage} = useIntl();

// HTML attribute
<input title={formatMessage({id: "form-field-title"})} />

// Object value
const options = items.map((item) => ({
	id: item.id,
	label: formatMessage({id: `feature-item-${item.type}`}),
}));

// aria label
<IconButton aria-label={formatMessage({id: "actions-delete-label"})}>
```

**The rule:** if it's JSX content, use `<FormattedMessage>` or `<Translation>`. If it's a string value being passed somewhere, use `formatMessage`.

---

## Dynamic Values and Plurals

Use ICU MessageFormat for pluralisation and variable interpolation. Define the pattern in `en-US.json`:

```json
{
  "activities-schedule": "{itemCount, plural, one {behavior scheduled} other {behaviors scheduled}}"
}
```

Pass values via the `values` prop or second argument:

```tsx
// FormattedMessage with values
<FormattedMessage id="activities-schedule" values={{itemCount: count}} />

// formatMessage with values
formatMessage({id: "activities-schedule"}, {itemCount: count});
```

## Type Safety

Always type translation props as `LanguageKeys`, never as `string`:

```tsx
// Correct
type MyComponentProps = {
	labelTranslationId: LanguageKeys;
	valueTranslationId?: LanguageKeys;
};

// Wrong — loses type safety, allows invalid keys
type MyComponentProps = {
	labelTranslationId: string;
};
```

---

## Language Files

- `en-US.json` — the master file, all keys must exist here
- `es-MX.json` — partial Spanish translation; missing keys fall back to English automatically
- `te-ST.json` — test language file; merges on top of English so untranslated keys remain readable

Never add new keys to `es-MX.json` or `te-ST.json` directly. New keys belong in `en-US.json` only.
